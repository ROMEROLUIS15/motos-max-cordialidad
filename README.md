# MotoWorkshop SaaS

[![CI](https://github.com/ROMEROLUIS15/motos-max-cordialidad/actions/workflows/ci.yml/badge.svg)](https://github.com/ROMEROLUIS15/motos-max-cordialidad/actions/workflows/ci.yml)

**🌐 English** · [Español](#motoworkshop-saas-español)

> 📝 **Note for reviewers:** the detailed technical docs (Architecture, ADRs, Security, Runbook under `docs/`) are kept in **Spanish at the client's request**, so the workshop owner can read them. This README is provided in both English and Spanish.

Multi-tenant SaaS for motorcycle workshops, with AI assistance over WhatsApp. Work orders, inventory, motorcycle sales, customer CRM and conversational support, all isolated per workshop (tenant).

---

## Features

- **Work orders** — vehicle reception, service lines, parts, photo evidence, state machine (pending → in progress → waiting for parts → completed → delivered), full per-vehicle history.
- **Inventory** — parts with per-branch stock (physical / reserved / available), entry/exit/adjustment/transfer movements, low-stock alerts.
- **Quotes and payments** — versioned quotes with customer approval, payment recording, PDF generation.
- **Motorcycle sales** — unit inventory (new/used), sale orders with a payment plan, sales-contract PDF, sales dashboard.
- **Customer CRM** — per-customer vehicle and order history, home service.
- **WhatsApp conversational assistant** — automated support for customers (RouterAgent) and for workshop administration (AgentAdmin), with periodic reports and stock alerts generated by AI.
- **Multi-tenant** — strict per-workshop isolation at the data and roles/permissions level, with action auditing.
- **Installable PWA**, light/dark theme, in-app notifications (the client refreshes on a timer and pauses on a hidden tab; the API also exposes a WebSocket gateway, see [ADR-012](docs/ADR.md)).

## Architecture

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Browser     │────▶│  Cloudflare      │────▶│  Render         │
│  (Next.js)  │     │  Pages (Next.js) │     │  NestJS API     │
└─────────────┘     └──────────────────┘     ─────┬──────────────┘
                                                  │
                    ┌──────────────────────────────┤
                    │                              │
                    ▼                              ▼
           ┌──────────────┐              ┌──────────────────┐
           │  Neon        │              │  Render          │
           │  PostgreSQL  │              │  FastAPI Agents  │
           └──────────────┘              └──────────────────┘
                    ▲                              │
                    │                              │
                    │                     ┌────────┴──────────┐
                    │                     │  Redis + R2       │
                    └─────────────────────┘                   │
                                                              │
                                                              ▼
                                                    ┌──────────────────┐
                                                    │  WhatsApp        │
                                                    │  Cloud API       │
                                                    └──────────────────┘
```

The API follows hexagonal architecture (domain → application → infrastructure → presentation) with dependency injection via ports. The agents microservice is a standalone Python process that talks to the API over short-lived service JWTs. See [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) (Spanish) for the full detail.

pnpm-workspaces monorepo with 4 packages:

| Package                | Path              | Role                            |
| ---------------------- | ----------------- | ------------------------------- |
| `@motoworkshop/api`    | `apps/api/`       | NestJS REST API (hexagonal)     |
| `@motoworkshop/web`    | `apps/web/`       | Next.js frontend                |
| `@motoworkshop/agents` | `apps/agents/`    | Python microservice (LangGraph) |
| `@motoworkshop/types`  | `packages/types/` | Shared TS types                 |

## Installation

```bash
# 1. Clone
git clone https://github.com/ROMEROLUIS15/motos-max-cordialidad.git
cd motos-max-cordialidad

# 2. Set up environment files
# Option A: automatic script (Windows)
setup-env.bat

# Option B: automatic script (Linux/Mac)
chmod +x setup-env.sh
./setup-env.sh

# Option C: manual (copy every file)
cp .env.local.example .env.local
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env.local
cp apps/agents/.env.example apps/agents/.env

# 3. Edit the .env files with your credentials
# - .env.local (general config)
# - apps/api/.env (NestJS API)
# - apps/web/.env.local (Next.js frontend)
# - apps/agents/.env (Python agents)

# 4. Install Node dependencies
# On Windows bcrypt needs build tools; if it fails, use --ignore-scripts:
pnpm install --ignore-scripts
# On macOS/Linux (or with build tools available):
pnpm install

# 5. Python (agents)
cd apps/agents
uv sync
cd ../..

# 6. Local infrastructure (Docker)
docker compose up -d

# 7. Database
pnpm --filter @motoworkshop/api db:generate
pnpm --filter @motoworkshop/api db:migrate
pnpm --filter @motoworkshop/api db:seed
```

## Deployment

The three services deploy only when the CI pipeline is green — there is no direct auto-deploy from Render or Cloudflare on push. See [docs/RUNBOOK.md](docs/RUNBOOK.md) (Spanish) for the full production runbook (per-service variables, rollback, incident diagnosis).

| Service       | Provider         | Trigger                                                |
| ------------- | ---------------- | ------------------------------------------------------ |
| NestJS API    | Render (Docker)  | Orchestrated from CI after green checks                |
| Python Agents | Render (Docker)  | Orchestrated from CI, only if `apps/agents/**` changed |
| Web Next.js   | Cloudflare Pages | Orchestrated from CI (`wrangler` via CLI flags)        |
| Database      | Neon             | Serverless PostgreSQL                                  |
| Storage       | Cloudflare R2    | S3-compatible                                          |

# MotoWorkshop SaaS (Español)

[English](#motoworkshop-saas) · **🌐 Español**

SaaS multi-tenant para talleres de motocicletas, con asistencia de IA vía WhatsApp. Gestión de órdenes de trabajo, inventario, venta de motocicletas, CRM de clientes y comunicación conversacional, todo aislado por taller (tenant).

---

## Funcionalidades

- **Órdenes de trabajo** — recepción de vehículo, líneas de servicio, repuestos, evidencia fotográfica, máquina de estados (pendiente → en progreso → esperando repuestos → completada → entregada), historial completo por vehículo.
- **Inventario** — repuestos con stock por sucursal (físico / reservado / disponible), movimientos de entrada/salida/ajuste/transferencia, alertas de stock bajo.
- **Cotizaciones y pagos** — cotizaciones versionadas con aprobación del cliente, registro de pagos, generación de PDF.
- **Venta de motocicletas** — inventario de unidades (nuevas/usadas), órdenes de venta con plan de pago, contrato de compraventa en PDF, dashboard de ventas.
- **CRM de clientes** — historial de vehículos y órdenes por cliente, servicio a domicilio.
- **Asistente conversacional por WhatsApp** — atención automática a clientes (RouterAgent) y a la administración del taller (AgentAdmin), con reportes periódicos y alertas de stock generados por IA.
- **Multi-tenant** — aislamiento estricto por taller a nivel de datos y de roles/permisos, con auditoría de acciones.
- **PWA instalable**, tema claro/oscuro, notificaciones in-app (el cliente refresca por temporizador y pausa en pestaña oculta; la API expone además un gateway WebSocket, ver [ADR-012](docs/ADR.md)).

## Calidad

El repositorio tiene un pipeline de CI que bloquea el merge y el deploy hasta que todo pasa: typecheck, tests con piso de cobertura, lint, tests del microservicio Python, e2e de API y de Web, escaneo de secretos y auditoría de dependencias. Ver [docs/RUNBOOK.md](docs/RUNBOOK.md#pipeline-de-cicd) para el detalle completo del pipeline.

```bash
pnpm typecheck              # TypeScript (API + Web)
pnpm lint                   # ESLint (API + Web)
pnpm test                   # Tests (API Jest + Web Vitest)
pnpm -r test:ci             # Tests en modo CI (con cobertura)

# API — end-to-end contra Postgres/Redis reales
pnpm --filter @motoworkshop/api test:e2e

# Python
cd apps/agents
uv run ruff check src tests
uv run mypy src
uv run pytest
```

## Despliegue

Los tres servicios se despliegan únicamente cuando el pipeline de CI está en verde — no hay auto-deploy directo de Render ni de Cloudflare al hacer push. Ver [docs/RUNBOOK.md](docs/RUNBOOK.md) para el runbook completo de producción (variables por servicio, rollback, diagnóstico de incidentes).

| Servicio       | Proveedor        | Disparador                                           |
| -------------- | ---------------- | ---------------------------------------------------- |
| API NestJS     | Render (Docker)  | Orquestado desde CI tras checks verdes               |
| Agentes Python | Render (Docker)  | Orquestado desde CI, solo si cambió `apps/agents/**` |
| Web Next.js    | Cloudflare Pages | Orquestado desde CI (`wrangler` vía flags de CLI)    |
| Base de datos  | Neon             | PostgreSQL serverless                                |
| Storage        | Cloudflare R2    | S3-compatible                                        |

## Contribuir

`main` es la rama protegida: requiere pull request y que los checks del pipeline de CI pasen antes de mergear. No se hace force-push ni se saltan los hooks locales (pre-commit escanea secretos, pre-push corre typecheck y tests).
